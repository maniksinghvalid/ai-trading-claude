<objective>
Add magic-link email auth + per-user session isolation (slice 8, AUTH-01): a one-time signed
magic link issues a 24h JWT on click; `/chat`, `/chat/stream`, and `/sessions` require
`Authorization: Bearer <jwt>`; `session_store` gains a `user_id` column and `list_sessions`
filters to the current user.

Purpose: Phase 1 is single-user with no auth. Phase 2 makes it multi-user with strict isolation:
User A must never see User B's sessions. This plan establishes the `user_id` data-model change
that 02-04 (Postgres), 02-05 (UserBudget), and 02-06 (per-user session list) all build on.

Output: `auth.py`, `routes/auth.py`, JWT middleware/dependency, a `user_id` column on `Turn`,
user-scoped session queries, and a minimal frontend login page that stores the JWT.
</objective>

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-03-01 | Spoofing | JWT forgery | mitigate | HS256 signed with jwt_secret; decode verifies signature + exp; reject on failure |
| T-02-03-02 | Elevation/IDOR | cross-user session access | mitigate | history()/list_sessions() filter by user_id from the token, not from the request body |
| T-02-03-03 | Info disclosure | email-provider errors / secret | mitigate | Wrap send; generic 503; secret in config from env, never in response |
| T-02-03-04 | Tampering | magic-link replay | mitigate | Short-TTL signed one-time token; verify_magic_token rejects expired/tampered |
| T-02-03-SC | Tampering | PyJWT + email SDK installs | mitigate | Blocking legitimacy gate (checkpoint above) verifies pypi.org before install |
