## Directory Purposes

**`trade/`:**
- Purpose: Single orchestrator skill that routes `/trade <command>` to matching sub-skill
- Contains: Only `SKILL.md` (the main entry point)
- Key file: `trade/SKILL.md` — 19-command routing table, consolidated scoring methodology

**`skills/`:**
- Purpose: Self-contained implementations of each `/trade` command
- Contains: 18 directories, each with a single `SKILL.md` file (no subdirectories within skills)
- Pattern: `skills/<command>/SKILL.md` where `<command>` matches the skill name (e.g., `trade-analyze`, `trade-quick`)
- Special notes:
  - `trade-analyze/SKILL.md` is the largest; implements 3-phase flow + agent coordination
  - `trade-routine/SKILL.md` coordinates portfolio sweep, escalation budgeting, options overlay
  - `trade-recall/SKILL.md` queries memory layer and renders citations
  - All others are single-purpose (standalone ticker analysis, portfolio ops, etc.)

**`agents/`:**
- Purpose: 5 subagent prompts invoked **only** by `/trade analyze` in Phase 2
- Contains: 5 Markdown files (not directories; no nesting)
- Files: `trade-technical.md`, `trade-fundamental.md`, `trade-sentiment.md`, `trade-risk.md`, `trade-thesis.md`
- Pattern: Each agent has YAML frontmatter (`name:`, `description:`) and a `**Weight:**` header documenting its scoring weight
- Special notes:
  - Agents receive `DISCOVERY_BRIEF` from Phase 1 as context
  - Each returns a structured score (0-100) broken into 5 subscores
  - Methodology should be kept consistent with corresponding standalone skill (if one exists)

**`scripts/`:**
- Purpose: Shared utilities (Python scripts, shell helpers)
- Key files:
  - `trade_memory.py` (producer) — ingest reports into Pinecone; query interface
  - `trade_schemas.py` — **SINGLE SOURCE OF TRUTH** for Pinecone record schema (fields, types, enums); vendored into proxy
  - `trade_scoring.py` — scoring weights, grade/signal lookup (referenced in prose by all files that score)
  - `generate_trade_pdf.py` — PDF renderer
  - `sync_proxy_schemas.sh` — copies `trade_schemas.py` to `proxy/_lib/trade_schemas.py` (diff gate D.17 enforces this)

**`proxy/`:**
- Purpose: Separate Vercel serverless function that holds Pinecone API key
- Contains: Flask-like Python app with 5 endpoints, validation logic, Postman collection
- Key files:
  - `proxy/app.py` — main entry point (Vercel function handler)
  - `proxy/api/*.py` — endpoint implementations (upsert, query, list, fetch, delete)
  - `proxy/_lib/validate.py` — re-validates payloads against schema (same schema as producer)
  - `proxy/_lib/trade_schemas.py` — vendored copy of `scripts/trade_schemas.py` (must match via sync script)
  - `proxy/vercel.json` — deployment config
  - `proxy/requirements.txt` — separate from root (Vercel has its own dependency install)
- Special notes:
  - Proxy has its own root directory and can be deployed independently
  - Never hardcode API keys in this directory; use Vercel env vars
  - Schema validation is re-done here (belt-and-suspenders approach)

**`plan/`:**
- Purpose: Multi-slice implementation plans and design docs
- Files: `portfolio-routine-and-vector-memory.md` (main 9-slice plan), `trading-chatbot.md`, `path-d-formalization-*.md`, etc.
- Special notes: These are reference docs for ongoing development; not source code

**.`planning/codebase/`:**
- Purpose: Generated codebase analysis documents (ARCHITECTURE.md, STRUCTURE.md, etc.)
- Generated by: `/gsd-map-codebase` command (outside this codebase's control)
- Should not be manually edited; overwritten on re-analysis

## Naming Conventions

**Files:**
- **Skill directories**: `skills/trade-<command>/` (e.g., `skills/trade-analyze/`, `skills/trade-quick/`)
- **Skill contents**: Always `SKILL.md` (no other files in skill directories)
- **Agent files**: `agents/trade-<dimension>.md` (e.g., `agents/trade-technical.md`, `agents/trade-risk.md`)
- **Output reports (CWD)**: `TRADE-<REPORT_TYPE>-<TICKER>[-<ts>].md`
  - Examples: `TRADE-ANALYSIS-AAPL.md`, `TRADE-QUICK-MSFT.md`, `TRADE-OPTIONS-TSLA-20260608-143022.md`
  - REPORT_TYPE values: ANALYSIS, QUICK, TECHNICAL, FUNDAMENTAL, SENTIMENT, RISK, THESIS, SECTOR, COMPARE, OPTIONS, EARNINGS, THESIS
  - Timestamp `-<ts>` (optional) used when multiple reports of same type per ticker (e.g., OPTIONS daily overlay)
- **Routine output**: `TRADE-ROUTINE-<ISO8601-ts>.md` (e.g., `TRADE-ROUTINE-20260608-143022.md`)
- **Watchlist output**: `TRADE-WATCHLIST.md`
- **Portfolio output**: `TRADE-PORTFOLIO.md`
- **Holdings cache**: `~/.claude/trade/TRADE-HOLDINGS.md`
- **PDF output**: `TRADE-REPORT.pdf`

**Directories:**
- Lowercase, hyphenated: `skills/`, `agents/`, `proxy/`, `plan/`
- Skill subdirs: `skills/trade-<command>/` (matches the skill name)

**Python/Shell scripts:**
- Snake_case: `trade_memory.py`, `trade_scoring.py`, `trade_schemas.py`, `generate_trade_pdf.py`
- Shell scripts: Same convention: `sync_proxy_schemas.sh`

**YAML Frontmatter Fields (Report Metadata):**
- `trade_report: true` — Always present; signals this is an ingestion-eligible report
- `schema_version: 1` — Pinecone schema version; bumped only on breaking changes
- `ticker: <TICKER>` — Stock symbol (uppercase)
- `company: <NAME>` — Full company name
- `report_type: <TYPE>` — One of: ANALYSIS, QUICK, TECHNICAL, FUNDAMENTAL, SENTIMENT, RISK, THESIS, SECTOR, COMPARE, OPTIONS, EARNINGS, PORTFOLIO, HOLDINGS, ROUTINE (uppercase, singular)
- `generated_at: <ISO8601>` — Timestamp with timezone (e.g., `2026-06-08T14:30:00-07:00`)
- `composite_score: <int>` — 0-100 (only ANALYSIS reports have this; QUICK has no numeric score)
- `signal: <ENUM>` — One of: STRONG BUY, BUY, HOLD, NEUTRAL, CAUTION, AVOID (uppercase, spaces)
- `grade: <ENUM>` — One of: A+, A, B, C, D, F (uppercase)
- `technical_score: <int>` — Dimension scores (only ANALYSIS; optional for other types)
- `fundamental_score: <int>`
- `sentiment_score: <int>`
- `risk_score: <int>` — **INVERTED**: higher = lower risk
- `thesis_score: <int>`
- `source_path: <path>` — Path to original report file (set by memory producer during ingest)
- `run_id: <id>` — Routine execution ID (set by routine, format: `routine-<YYYYMMDD-HHMM>-<6hex>`)

**Python Variable Names (in scripts):**
- Snake_case: `run_id`, `max_escalations`, `prior_signal`, `escalations_used`, `ticker_results`
- Class names: PascalCase (e.g., `PineconeClient`, `SchemaValidator`)
- Constants: UPPERCASE with underscores (e.g., `DEFAULT_MAX_ESCALATIONS`, `PINECONE_NAMESPACE`)

## Where to Add New Code

**New Command (e.g., `/trade new-thing <ticker>`):**
1. Create directory: `skills/trade-new-thing/`
2. Create skill: `skills/trade-new-thing/SKILL.md` with YAML frontmatter:
   ```yaml
   ---
   name: trade-new-thing
   description: One-line description of what the command does
   ---
   ```
3. Add routing entry in `trade/SKILL.md` (Command Reference table), including output filename pattern
4. Add skill name to `SKILLS=(...)` array in both `install.sh` and `uninstall.sh`
5. Update README "All 19 Commands" section and project structure documentation
6. If the skill writes a report file with YAML frontmatter, add schema to `scripts/trade_schemas.py` (for `report_type: NEW_THING`)

**New Dimension/Subagent (for `/trade analyze` Phase 2 fan-out):**
1. Create agent: `agents/trade-new-dimension.md` with YAML frontmatter and `**Weight:** X%` header
2. Update `trade/SKILL.md` scoring methodology table to include new dimension weight (and adjust existing weights to sum to 100%)
3. Update `skills/trade-analyze/SKILL.md` Phase 2 agent list to launch new agent
4. Update Phase 3 composite formula in `trade-analyze` and `trade/SKILL.md`
5. Update `scripts/trade_scoring.py` weights and grade/signal formula if weights changed
6. Add dimension-specific score field to `scripts/trade_schemas.py` (e.g., `new_dimension_score: int`)
7. **CRITICAL**: Update **4 files** with new weights (README, orchestrator, analyze skill, new agent + all 4 existing agents)
8. Sync proxy schema: `bash scripts/sync_proxy_schemas.sh`
9. Update all documentation (README cross-file contract section)

**New Skill-to-Agent Coordination Pattern:**
1. If you want a standalone `/trade new-skill` that also feeds into `/trade analyze` (like technical, fundamental, etc.):
   - Create standalone skill: `skills/trade-new-skill/SKILL.md` (full methodology)
   - Create subagent: `agents/trade-new-skill.md` (same methodology, receives `DISCOVERY_BRIEF`)
   - Keep methodology synchronized between both files
   - Add skill to install.sh (but not to agent list — agents are only for analyze)
   - See `agents/trade-technical.md` vs `skills/trade-technical/SKILL.md` for the pattern

**Utility Scripts:**
- Add Python scripts to `scripts/` directory
- If script is a producer (writes to Pinecone), validate with `scripts/trade_schemas.py` before upsert
- If script is CLI-invokable, add `if __name__ == "__main__":` block and document usage in comments
- See `scripts/trade_memory.py` (producer) and `scripts/generate_trade_pdf.py` (renderer) for patterns

**API Endpoints (Proxy):**
1. Create handler: `proxy/api/<endpoint_name>.py` with function matching Vercel pattern
2. Register in `proxy/app.py` (add route + handler dispatch)
3. Add payload schema to `proxy/_lib/trade_schemas.py` (request/response types)
4. Add validation in `proxy/_lib/validate.py`
5. Add test in `proxy/postman/pinecone-proxy.postman_collection.json`

**Configuration & Contracts:**
- Never edit `install.sh` / `uninstall.sh` without adding your skill to both SKILLS and AGENTS arrays (or it won't install/uninstall cleanly)
- Never add a report_type without adding its schema to `scripts/trade_schemas.py` and syncing proxy
- Never change scoring weights without updating all 4 weight-declaring files (see CLAUDE.md cross-file contract)
- Never rename frontmatter fields without bumping `schema_version` in `scripts/trade_schemas.py`

## Special Directories

**`~/.claude/trade/`:**
- Purpose: Cache directory for user-specific state (holdings, prior scores, etc.)
- Generated: Yes (by `/trade holdings` and `/trade routine`)
- Committed: No (user-specific; lives on user's machine after `install.sh`)
- Key file: `TRADE-HOLDINGS.md` — portfolio holdings cache (read from Drive, written locally for fallback)

**`.planning/codebase/`:**
- Purpose: Archive of generated codebase maps (ARCHITECTURE.md, STRUCTURE.md, CONVENTIONS.md, TESTING.md, CONCERNS.md)
- Generated: Yes (by `/gsd-map-codebase` command, external to this repo)
- Committed: Yes (tracked in git; provides continuity across Claude Code sessions)

**`proxy/`:**
- Purpose: Separate Vercel project root
- Committed: Yes (tracked in main repo, but can be deployed independently)
- Deployment: Via Vercel CLI or MCP (`npm i -g vercel && vercel --prod`)
- Important: `proxy/requirements.txt` is separate from root; Vercel installs independently

```
    │
    ├─► /trade quick <ticker>

    ├─► /trade routine [flags]

    │           │     ├─► trade_memory.py recommend-tier
    │           │     ├─► Dispatch /trade analyze OR quick
    │           │     └─► (optional) /trade options overlay
    │           ├─► Ingest results: trade_memory.py ingest
    │           └─► Write TRADE-ROUTINE-<ts>.md (+ optional cloud delivery)
    │
    ├─► /trade recall "<query>" [args]
    │     └─► skills/trade-recall/SKILL.md
    │           ├─► trade_memory.py doctor (health check)
    │           ├─► trade_memory.py query (semantic search)
    │           └─► Render cited findings
    │
    └─► /trade <other-command>
          └─► skills/trade-<other>/SKILL.md
                └─► Specific logic (technical, fundamental, etc.)

Memory Ingest Flow:
  Report file (TRADE-*.md in CWD)
    │
    └─► scripts/trade_memory.py ingest
          ├─► Parse YAML frontmatter
          ├─► Validate against scripts/trade_schemas.py
          ├─► (Local): Upsert to Pinecone directly
          │   OR
          ├─► (Cloud): POST /upsert to proxy (Vercel)
          │     └─► proxy/_lib/validate.py re-validates
          │     └─► proxy/api/upsert.py forwards to Pinecone
          └─► Archive metadata

Memory Query Flow:
  /trade recall "<query>"
    │
    └─► scripts/trade_memory.py query
          ├─► (Local): Query Pinecone directly
          │   OR
          ├─► (Cloud): POST /query to proxy
          │     └─► proxy/api/query.py forwards to Pinecone
          └─► Return hit list (metadata + text chunks)
```
